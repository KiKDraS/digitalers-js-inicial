API REST

El término "Transferencia de Estado Representacional" (REST) representa un conjunto de características de diseño de arquitecturas software que aportan confiabilidad, eficiencia y escalibilidad a los sistemas distribuidos. Un sistema es llamado RESTfull cuando se ajusta a estas características.
La idea básica de REST es que un recurso, e.j. un documento, es transferido con su estado y su relaciones (hipertexto) mediante formatos y operaciones estandarizadas bien definidas.
Como HTTP, el protocolo estándar de la Web, también transfiere documentos e hipertexto, las APIs HTTP a veces son llamadas APIs RESTfull, servicios RESTful, o simplemente servicios REST, aunque no se ajusten del todo a la definición de REST.

Verbos REST

  • GET -> 'Obtener dato'
  • POST -> 'Enviar Dato'
  • PUT -> 'Modificar Dato'
  • DELETE -> 'Borrar Dato'

CORS y Headers

El Intercambio de Recursos de Origen Cruzado (CORS) es un mecanismo que utiliza cabeceras HTTP adicionales para permitir que un user agent (en-US) obtenga permiso para acceder a recursos seleccionados desde un servidor, en un origen distinto (dominio) al que pertenece. Un agente crea una petición HTTP de origen cruzado cuando solicita un recurso desde un dominio distinto, un protocolo o un puerto diferente al del documento que lo generó.
Un ejemplo de solicitud de origen cruzado: el código JavaScript frontend de una aplicación web que es localizada en http://domain-a.com utiliza XMLHttpRequest para cargar el recurso http://api.domain-b.com/data.json.
El trabajo con los Headers nos permite manipular la configuración para evitar este tipo de errores. Cabe destacar que saltarse el protocolo CORS no se considera una practica segura y, por tanto, debe evitarse.
Los Headers son usualmente utilizados para configurar el tipo de dato que se va a estar enviando y recibiendo.
No todos los Headers son modificables.

Single Page Application (SPA)

En una Single Page Application (SPA) todos los códigos (HTML, CSS, JS) se cargan por única vez durante la carga inicial del sitio. La vista es modificada dinámicamente de acuerdo a las distintas interacciones realizadas por el usuario.
Una SPA genera una distintiva mejora en la velocidad de navegación pero su aplicación implica lidiar con algunos problemas de SEO, implementar una navegación híbrida mediante el uso de las API's History y Location, y el esfuerzo extra que conlleva mantener el estado del sitio.

Posibles Soluciones a Problemas de SEO

API's History y Location

Las API's History y Location manejan el historial de navegación (history) y la URL de navegación (location).
Usadas en combinación, nos permiten realizar el 'ruteo' de las SPA

Location

  • location.origin => ruta de origen(root - dominio)
  • location.href => url actual
  • location.hash => ancla (#) actual
  • location.search => datos pasados por parámetro
  • location.pathname => ruta del nombre del archivo

History

  • history.length => array que contiene las páginas visitadas
  • history.forward(x) => x páginas hacia adelante
  • history.back(x) => x páginas hacia atrás
  • history.go(x) => -x hacia atras; x hacia adelante
  • history.pushState(estado,titulo,url) => genera un nuevo historial de navegación
